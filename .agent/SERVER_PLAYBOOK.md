# 服务器运行手册

本文件描述 OpenClaw 在服务器上的无人值守运行方式。目标不是让 agent 无限工作，而是让它以稳定、可恢复、可停止的节奏推进一个个小任务。

## 运行目标

服务器上的 OpenClaw 每次唤醒只推进至多一个安全任务，或把一个已在进行中的任务推进一个检查点。

不要在单次唤醒中连续吞掉多个任务，也不要为了“效率”牺牲可审查性。

## 推荐唤醒节奏

推荐从以下节奏开始：

- 工作日每 2 到 4 小时唤醒一次
- 非工作时间降低频率，例如每天 1 到 2 次

如果当前任务主要依赖人工反馈、PR 审查或外部环境，应该降频，而不是无意义重试。

## 每次唤醒的标准流程

1. 拉取最新仓库状态并检查工作区是否干净。
2. 如果当前分支对应某个 `in_progress` 任务，先处理该任务。
3. 如果没有进行中的任务，从 GitHub Issues 选择一个 `status:ready` 任务：
   `gh issue list --repo songxychn/knife4j-next --label agent-task --label status:ready --state open`
4. 判断任务是否符合 `.agent/AUTONOMY_POLICY.md` 中的安全条件。
5. 决定直接执行还是委派给 worker。
6. 完成最小改动后，运行最窄相关验证。
7. 需要时启动 reviewer。
8. 在对应 GitHub Issue 评论里记录进度、验证结果和下一步（不再写 .agent/TASKS.md 或 .agent/PROGRESS.md，这两个文件已冻结）。
9. 满足条件时创建或更新 PR。
10. 等待 PR CI 终态：`gh pr checks <N> --watch`。全绿后给 issue 加 `status:review` label；CI 红时同分支修复再等（单次唤醒最多 2 轮修复尝试）。
11. 停止本次运行，不继续吞下一个新任务。

## 任务选择策略

默认按以下顺序选择：

1. `in_progress` 且未完成的任务
2. 依赖已满足的 `ready` 任务
3. 验证成本较低、范围较窄的 `ready` 任务
4. 文档或 repo 维护类任务
5. front-core 这类有直接 test/lint/build 闭环的任务

不要优先选择：

- 需要产品判断的任务
- 范围不清的 Java 兼容性任务
- 依赖尚未满足的任务
- 预计会触发大规模依赖安装或环境变更的任务

## 什么时候创建分支

当一个任务被正式选中准备执行时，立即使用任务中声明的 `branch` 创建或切换分支。

如果分支已存在：

- 优先继续已有分支
- 如果分支状态和任务状态不一致，通过 GitHub Issue label 和评论对齐

## 什么时候派 Worker

满足以下任一条件时，优先派 worker：

- 为了开始改动需要阅读很多文件
- 同一个任务可拆成互不重叠的探索切片
- coordinator 如果亲自做会积累太多实现细节
- 可以把一次探索压缩成一个 handoff

不要为了“显得高级”而强行派 worker。任务很小时，coordinator 直接做更稳。

## 什么时候派 Reviewer

满足以下任一条件时，PR 前应派 reviewer：

- 实现由 worker 完成
- Java 兼容性行为发生变化
- 改动涉及多个模块
- 验证曾失败后才通过
- coordinator 对 diff 的回归风险没有足够把握

## PR 触发条件

只有满足以下条件时才应创建或更新 PR：

- 至少完成该任务要求的最窄验证（含 `./scripts/test-*.sh`，不只用 `tsc` / `vite build` 单步替代）
- 在对应 GitHub Issue 评论里已记录摘要、验证和下一步
- 没有未说明的高风险残留
- 创建 PR 后必须等 CI 全绿，才把 issue 切为 `status:review`（见 .agent/RUNBOOK.md “PR 后 CI 验证”节）

PR 描述应至少包含：

- task id
- 改动范围
- 运行过的验证命令
- 当前已知风险
- 是否需要人工决策

## 失败处理策略

### 实现失败

如果实现中卡住：

- 记录 blocker
- 判断 blocker 是任务内问题还是任务外问题
- 任务内问题可在本次唤醒内再尝试一次
- 如果第二次仍失败，停止并记录，不要无限试错

### 验证失败

如果验证失败：

- 优先判断是否是当前改动引起
- 如果是当前改动引起，可做一次针对性修复并重跑
- 如果第二次仍失败，将任务记为 `blocked` 或保持 `in_progress` 并明确 blocker
- 如果失败超出当前任务范围，不要继续扩散处理，创建后续任务

### 环境失败

如果是网络、依赖源、CI 凭据、本地环境损坏等环境问题：

- 不要把环境问题当成代码问题反复重试
- 在对应 GitHub Issue 评论里记录失败命令和环境现象
- 给 issue 加 `status:blocked` label

## 重试规则

默认重试上限：

- 同一次唤醒内，对同一实现问题最多 2 次尝试
- 同一次唤醒内，对同一验证问题最多 2 次运行
- 跨多次唤醒，如果同一 blocker 连续出现 2 次，应等待人工处理

目标是防止 OpenClaw 在服务器上无意义消耗时间。

## 停止条件

出现以下任一情况时，本次运行应停止：

- 已完成一个任务的可审查提交
- 已记录 blocker，等待人工
- 已完成一次失败后的合理重试
- 已创建或更新 PR，CI 全绿（或 CI 红且已用完重试上限），且没有新的安全动作可做
- 工作区出现未预期冲突或状态异常

## 状态写回规则

每次运行结束前，至少写回：

- 当前 task id
- 当前分支
- 任务状态变化
- 关键验证结果
- 下一步
- blocker，如有

持久状态只写入 `.agent/`，不要把关键结论只留在服务日志里。

## 推荐通知时机

建议在以下时机通知维护者：

- 新 PR 已创建
- 任务进入 `blocked`
- reviewer 给出 `block`
- 发现需要人工决策的兼容性或发布问题

## 不要做的事

- 一次 heartbeat 连做多个任务
- 为了“清空队列”而降低验证标准
- 在失败后无上限重试
- 未经批准处理 release 或 secrets
- 在没有任务记录的情况下临时起意改动仓库
